# 第二阶段学习总览：Tool Calling / Structured Output / MCP

## 文档目标

第二阶段的核心目标是理解：

> AI 如何调用外部能力，并把模型输出变成可被程序稳定消费的数据。

第一阶段主要解决“如何和模型对话”。第二阶段开始进入 AI 应用工程的关键分水岭：

- 模型不再只返回自然语言
- 模型可以决定是否调用工具
- 工具执行结果会重新进入上下文
- 应用可以拿到结构化结果并继续执行后续业务逻辑

这也是 Agent、AI IDE、AI 办公助手、AI 数据分析工具、AI 自动化工作流的基础。

## 一张图理解第二阶段

```text
用户问题
  ↓
模型判断意图
  ↓
是否需要工具？
  ├─ 不需要：直接回答
  ↓
生成 tool call
  ↓
应用层校验参数
  ↓
执行工具函数 / 外部 API / 数据库
  ↓
把 tool result 发回模型
  ↓
模型生成最终回答
  ↓
结构化输出给前端或业务系统
```

这里要注意一个关键点：

> 模型不直接执行工具，真正执行工具的是你的应用代码。

模型只负责产生“我要调用哪个工具，以及参数是什么”的结构化请求。

## 需要建立的核心认知

### 1. Tool Calling 不是魔法

Tool Calling 本质是：

- 开发者把可用工具描述给模型
- 模型根据用户问题选择一个或多个工具
- 模型输出一段结构化的 tool call
- 应用层执行对应函数
- 函数结果再交给模型

模型本身不能查天气、不能访问数据库、不能调用公司接口。它只能通过你的应用获得这些能力。

---

### 2. Schema 是模型和程序之间的契约

自然语言适合给人看，不适合给程序稳定处理。

例如下面这种输出很难稳定解析：

```text
明天上海的天气大概是晴天，温度 23 到 29 度。
```

程序更需要：

```json
{
  "city": "上海",
  "date": "明天",
  "weather": "晴",
  "temperatureMin": 23,
  "temperatureMax": 29
}
```

这就是 Structured Output 的价值。

---

### 3. Agent 的最小形态是循环

一个简单 Agent 通常包含：

- 观察用户输入
- 思考是否需要工具
- 调用工具
- 观察工具结果
- 决定继续调用还是最终回答

可以抽象成：

```text
while not finished:
  ask model
  if model returns tool call:
    execute tool
    append tool result
  else:
    return final answer
```

复杂 Agent、LangGraph、Workflow，本质都是在这个循环上继续增强状态管理、分支控制、错误恢复和任务拆解能力。
