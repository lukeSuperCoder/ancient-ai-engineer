# Embedding 与相似度检索

## 文档目标

这份文档讲清楚 RAG 的第一个基础能力：

> 如何把文本变成可以检索的向量。

学完后，你应该能理解：

- Embedding 是什么
- 为什么文本可以用向量表示
- Similarity Search 是什么
- Cosine Similarity 如何理解
- RAG 为什么先检索再生成

---

## 1. Embedding 是什么

Embedding 可以理解为：

> 把一段文本转换成一组数字。

例如一句话：

```text
员工如何申请年假？
```

经过 Embedding 模型后，会变成类似这样的向量：

```text
[0.12, -0.38, 0.55, 0.09, ...]
```

真实向量通常有几百到几千维，不适合人直接阅读，但适合程序计算。

---

## 2. 为什么文本能变成向量

Embedding 模型在大量文本上训练过，它会把语义相近的文本映射到相近的向量位置。

例如：

```text
如何申请年假？
年假流程是什么？
休假审批在哪里提交？
```

这些句子字面不同，但语义接近，所以向量距离也会比较近。

再看另一个句子：

```text
服务器 CPU 使用率过高怎么办？
```

它和“年假流程”的主题差很远，向量距离就会更远。

---

## 3. RAG 为什么需要 Embedding

RAG 需要从大量资料中找到和用户问题相关的片段。

如果只用关键词搜索，会遇到问题：

- 用户说“请假”，文档写“休假”
- 用户说“报销流程”，文档写“费用审批”
- 用户问题很口语化，文档表达很正式

Embedding 检索可以按语义找内容，而不是只按字面匹配。

所以 RAG 的检索流程通常是：

```text
用户问题
  ↓
生成问题向量
  ↓
和知识库 Chunk 向量比较相似度
  ↓
返回最相似的几个 Chunk
```

---

## 4. Similarity Search 是什么

Similarity Search 就是相似度搜索。

它要解决的问题是：

> 给定一个查询向量，找到数据库中最相似的几个向量。

在 RAG 中：

- 查询向量来自用户问题
- 数据库向量来自文档 Chunk
- 检索结果是与问题最相关的文档片段

示例：

```text
问题：公司年假怎么申请？

检索结果 Top 3：
1. 年假申请流程说明
2. 员工休假审批制度
3. 请假系统操作手册
```

---

## 5. Cosine Similarity 如何理解

Cosine Similarity 常用于衡量两个向量方向是否接近。

可以先用直觉理解：

```text
两个向量方向越接近，语义越相似。
两个向量方向越偏离，语义越不相似。
```

取值通常可以理解为：

```text
接近 1：非常相似
接近 0：关系较弱
接近 -1：方向相反
```

在 RAG 中，我们更关心排序：

```text
相似度 0.86 的 Chunk
通常比相似度 0.42 的 Chunk
更适合放进 Prompt。
```

## 6. 一个最小检索例子

假设知识库有三个 Chunk：

```text
Chunk A：员工年假申请需要在 OA 系统提交。
Chunk B：服务器报警需要联系运维团队。
Chunk C：差旅报销需要上传发票和审批单。
```

用户提问：

```text
我想休年假，应该怎么操作？
```

Embedding 检索后，最可能返回：

```text
Chunk A：员工年假申请需要在 OA 系统提交。
```

因为“休年假”和“年假申请”语义最接近。

## 7. 工程中需要保存什么

一个 Chunk 入库时，通常要保存两类信息：

### 7.1 原始内容

```json
{
  "chunk_id": "chunk_001",
  "content": "员工年假申请需要在 OA 系统提交。"
}
```

### 7.2 向量和元数据

```json
{
  "embedding": [0.12, -0.38, 0.55],
  "document_id": "doc_001",
  "knowledge_base_id": "kb_hr",
  "page": 3,
  "source": "员工手册.pdf"
}
```

RAG 检索时用 `embedding` 找相似内容，回答时用 `content` 和 `source` 生成答案和引用。

## 8. 常见误区

### 误区一：Embedding 能直接回答问题

Embedding 只负责表示语义和计算相似度，不负责生成回答。

生成回答是 LLM 的工作。

---

### 误区二：相似度最高就一定正确

不一定。

相似度高只说明“语义接近”，不代表：

- 内容是最新的
- 用户有权限查看
- 文档本身是正确的
- Chunk 里包含完整答案

所以企业 RAG 还需要 Metadata Filter、Rerank 和 Citation。

---

### 误区三：只靠向量检索就够了

很多企业资料里有大量专有名词、编号、产品型号、人名、合同号。

这些内容用关键词搜索反而更准。

所以成熟 RAG 经常会使用 Hybrid Search：

> 向量检索 + 关键词检索。

## 10. 推荐实践

建议做一个最小实验：

1. 准备 5 到 10 段短文本
2. 给每段文本生成 Embedding
3. 给一个用户问题生成 Embedding
4. 计算问题和每段文本的相似度
5. 按相似度排序输出 Top 3

这个实验不需要先接入完整 RAG 系统。只要能看懂“语义相似的文本被排在前面”，就算掌握了 Embedding 检索的核心。
