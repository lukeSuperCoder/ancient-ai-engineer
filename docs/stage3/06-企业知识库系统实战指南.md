# 企业知识库系统实战指南

## 文档目标

这份文档把第三阶段知识串成一个项目：

> 从 0 到 1 设计一个企业 RAG 知识库系统。

学完后，你应该能理解：

- 企业知识库系统需要哪些模块
- 文档上传后如何处理
- 用户提问时如何检索和回答
- 多知识库、权限过滤、历史记录如何设计
- 最小可用版本应该先做什么

---

## 1. 项目目标

企业知识库系统要解决的问题是：

> 员工可以上传企业资料，然后像聊天一样提问，系统基于资料回答并给出来源。

最小可用版本需要支持：

- PDF 上传
- 文本解析
- 自动 Chunk
- Embedding 入库
- 向量检索
- Prompt 拼接
- 输出回答
- 来源引用

进阶版本再支持：

- 多知识库分类
- 权限过滤
- 历史记录
- 文档更新
- 检索评估
- Hybrid Search
- Rerank

---

## 2. 系统模块拆分

可以拆成六个模块：

```text
前端页面
  ↓
文档管理模块
  ↓
文档处理模块
  ↓
向量检索模块
  ↓
问答生成模块
  ↓
权限与历史模块
```

### 2.1 前端页面

负责：

- 上传文档
- 展示知识库列表
- 展示文档处理状态
- 提供聊天问答界面
- 展示答案和引用来源

### 2.2 文档管理模块

负责：

- 保存文档基本信息
- 记录上传人
- 记录所属知识库
- 记录处理状态

### 2.3 文档处理模块

负责：

- 解析 PDF 文本
- 清洗文本
- Chunk 切块
- 调用 Embedding 模型
- 写入向量数据库

### 2.4 向量检索模块

负责：

- 接收用户问题
- 生成问题向量
- 按权限和知识库过滤
- 检索 Top K Chunk

### 2.5 问答生成模块

负责：

- 组装 RAG Prompt
- 调用 LLM
- 要求模型基于资料回答
- 返回答案和来源

### 2.6 权限与历史模块

负责：

- 控制用户能访问哪些知识库
- 控制用户能检索哪些文档
- 保存问答记录
- 保存引用来源

## 3. 推荐数据表设计

### 3.1 知识库表

```text

- id
- name
- description
- owner_id
```

## 4. 文档上传流程

```text
用户上传 PDF
  ↓
保存文件
  ↓
创建 document 记录，状态为 uploaded
  ↓
启动处理任务，状态改为 processing
  ↓
解析 PDF 文本
  ↓
清洗文本
  ↓
按策略切 Chunk
  ↓
调用 Embedding
  ↓
写入 document_chunks
  ↓
document 状态改为 ready
```

如果处理失败：

```text
document 状态改为 failed
保存错误信息
前端展示失败原因
```

---

## 5. 用户问答流程

```text
用户输入问题
  ↓
确认用户可访问的知识库
  ↓
问题生成 Embedding
  ↓
向量检索 Top K Chunk
  ↓
按相似度和权限过滤
  ↓
组装 Prompt
  ↓
调用 LLM
  ↓
返回答案和 Citation
  ↓
保存问答历史
```

关键点：

> 权限过滤必须发生在检索阶段，不能只在前端隐藏。

## 6. RAG Prompt 模板

可以从这个模板开始：

```text
你是企业知识库助手。

请遵守以下规则：
1. 只能根据【知识库资料】回答。
2. 如果资料不足，请明确说明“知识库中没有找到相关依据”。
3. 不要编造制度、流程、数字或联系人。
4. 回答后列出引用来源。

【知识库资料】
{context}

【用户问题】
{question}
```

`context` 可以这样组织：

```text
【资料 1】
来源：员工手册.pdf，第 12 页
内容：年假申请需要在 OA 系统提交。

【资料 2】
来源：员工手册.pdf，第 13 页
内容：请假超过 3 天需要直属领导审批。
```

## 7. Citation 输出格式

建议让接口返回结构化数据：

```json
{
  "answer": "年假申请需要在 OA 系统提交。如果超过 3 天，需要直属领导审批。",
  "sources": [
    {
      "document_id": "doc_001",
      "filename": "员工手册.pdf",
      "page": 12,
      "section": "年假申请流程",
      "quote": "年假申请需要在 OA 系统提交。"
    },
    {
      "document_id": "doc_001",
      "filename": "员工手册.pdf",
      "page": 13,
      "section": "请假审批",
      "quote": "请假超过 3 天需要直属领导审批。"
    }
  ]
}
```

前端可以把来源展示成可点击列表。

## 8. 多知识库管理

多知识库的目标是把资料分类。

例如：

```text
HR 知识库
财务制度知识库
产品文档知识库
技术支持知识库
销售话术知识库
```

用户提问时可以：

- 只选择一个知识库
- 同时选择多个知识库
- 默认检索用户有权限的全部知识库

检索时必须带上：

```text
knowledge_base_id in selected_or_allowed_kbs
```

## 9. 权限过滤设计

权限过滤至少要考虑：

- 用户是否能访问这个知识库
- 用户是否能查看这个文档
- 用户是否能看到某些敏感分类

不要只在页面上隐藏知识库。

后端检索时必须限制：

```text
只在用户有权限的 Chunk 范围内做检索。
```

否则可能出现：

```text
用户看不到某个文档，但通过提问让 AI 把文档内容回答出来。
```

这是企业知识库的严重问题。

---

## 10. 历史记录设计

历史记录至少保存：

- 用户问题
- AI 回答
- 使用了哪些来源
- 检索到的 Chunk
- 创建时间

这些数据有三个价值：

- 用户可以回看对话
- 开发者可以排查错误
- 后续可以做评估和优化

建议保存结构化 sources，而不是只保存自然语言答案。

---

## 11. 最小可用版本开发顺序

建议按这个顺序实现：

1. 创建知识库和文档表
2. 支持上传 PDF
3. 解析 PDF 文本
4. 实现 Chunk 切块
5. 调用 Embedding 并入库
6. 实现向量检索 Top K
7. 拼接 Prompt 调用 LLM
8. 返回答案和来源
9. 保存问答历史
10. 增加知识库过滤
11. 增加权限过滤

不要一开始就做复杂 UI、Rerank 或多租户。

先跑通主链路：

```text
上传 → 入库 → 提问 → 检索 → 回答 → 引用
```

## 12. 测试用例建议

准备 10 到 20 个问题，覆盖不同情况：

```text
能直接回答的问题
需要多个 Chunk 才能回答的问题
知识库里没有答案的问题
容易被关键词误导的问题
不同知识库同名概念的问题
无权限文档相关的问题
```

每个问题记录：

```text
期望来源
期望回答要点
是否应该拒答
```

这样才能判断系统是不是真的变好。

## 13. 常见风险

### 风险一：PDF 解析质量差

PDF 里的表格、页眉页脚、换行可能影响 Chunk。

需要在入库前做文本清洗。

### 风险二：权限只做在前端

前端隐藏不等于后端安全。

检索接口必须做权限过滤。

### 风险三：来源引用不精确

如果 Citation 只显示文档名，不显示页码或章节，用户很难验证。

### 风险四：历史记录不保存来源

以后排查错误时，会不知道当时模型基于哪些资料回答。

## 15. 最终项目完成标准

当你的项目做到下面这些能力，就完成了第三阶段目标：

- 可以上传 PDF
- 可以解析并切块
- 可以生成 Embedding 并入库
- 可以按用户问题检索相关 Chunk
- 可以把 Chunk 拼进 Prompt
- 可以生成基于资料的回答
- 可以返回来源引用
- 可以按知识库分类检索
- 可以做基础权限过滤
- 可以保存问答历史

这时你已经具备独立开发企业级 RAG 系统基础版本的能力。
