# 向量数据库与索引原理

## 文档目标

这份文档讲清楚 RAG 的存储层：

> 向量数据库负责保存 Chunk 向量，并快速找出相似内容。

学完后，你应该能理解：

- 向量数据库解决什么问题
- pgvector 是什么
- Milvus / Pinecone 适合什么场景
- 向量索引为什么重要
- 企业知识库表结构应该怎么设计

---

## 1. 为什么需要向量数据库

Embedding 会把文本变成向量。

如果知识库里只有几十个 Chunk，可以直接在内存里逐个计算相似度。

但企业知识库可能有：

- 几万份文档
- 几百万个 Chunk
- 多个部门知识库
- 多个用户同时查询

这时需要一个专门的存储和检索系统。

向量数据库主要负责：

- 保存向量
- 保存原文和元数据
- 根据查询向量检索相似向量
- 支持 Top K 返回
- 支持元数据过滤

---

## 2. 普通数据库和向量数据库的区别

普通数据库擅长精确查询：

```sql
select * from documents where id = 'doc_001';
```

向量数据库擅长相似查询：

```text
找出和这个问题语义最相似的 5 个 Chunk。
```

两者不是替代关系，而是互补关系。

企业 RAG 通常同时需要：

- 关系型数据库保存用户、权限、文档记录
- 向量数据库保存 Chunk 向量和检索索引

---

## 3. pgvector 是什么

pgvector 是 PostgreSQL 的向量扩展。

它让 PostgreSQL 可以存储和检索向量。

适合：

- 中小型知识库
- 已经使用 PostgreSQL 的项目
- 希望减少系统组件数量
- 学习和原型开发

典型表结构：

```sql
create table document_chunks (
  id bigserial primary key,
  document_id text not null,
  knowledge_base_id text not null,
  content text not null,
  source text,
  page int,
  embedding vector(1536)
);
```

查询时可以按向量距离排序：

```sql
select content, source, page
from document_chunks
where knowledge_base_id = 'kb_hr'
order by embedding <=> query_embedding
limit 5;
```

这里的 `query_embedding` 表示用户问题的向量。

---

## 4. Milvus / Pinecone 是什么

Milvus 和 Pinecone 都是更专门的向量数据库。

### Milvus

Milvus 是开源向量数据库。

适合：

- 数据量较大
- 希望自建部署
- 团队有运维能力
- 需要更强的向量检索性能

### Pinecone

Pinecone 是托管向量数据库服务。

适合：

- 不想自己维护向量数据库
- 希望快速上线
- 可以接受云服务成本
- 更关注业务开发

### 如何选择

学习阶段建议：

```text
先用 pgvector 或本地轻量方案跑通 RAG。
数据量大、性能压力明确后，再考虑 Milvus / Pinecone。
```

## 5. 向量索引为什么重要

如果有 100 万个 Chunk，每次查询都逐个计算相似度，会很慢。

向量索引的作用是：

> 用更快的方式找到近似最相似的向量。

注意这里通常是“近似”，不是百分百精确。

工程上常见权衡是：

```text
检索更快
  ↔
结果可能略有误差
```

RAG 通常可以接受近似检索，因为后面还可以通过 Rerank 再精排。

---

## 6. Top K 是什么

Top K 表示返回最相似的 K 个结果。

例如：

```text
Top K = 5
```

表示检索出最相似的 5 个 Chunk。

K 太小：

- 可能漏掉答案
- 问题需要多段资料时不够用

K 太大：

- Prompt 变长
- 成本增加
- 噪声变多

常见起步值：

```text
Top K = 3 到 8
```

## 7. 企业知识库的数据模型

企业 RAG 通常至少需要三类表。

### 7.1 知识库表

```text
knowledge_bases
- id
- name
- description
- owner_id
- created_at
```

### 7.2 文档表

```text
documents
- id
- knowledge_base_id
- filename
- file_type
- status
- uploaded_by
- created_at
```

### 7.3 Chunk 表

```text
document_chunks
- id
- document_id
- knowledge_base_id
- content
- embedding
- page
- section
- metadata
- created_at
```

如果要做权限过滤，还需要用户、角色、部门、知识库授权关系。

---

## 8. Metadata Filter

Metadata Filter 是企业 RAG 非常重要的能力。

用户提问时，不应该检索所有文档，而应该先过滤范围。

例如：

```text
只检索 HR 知识库
只检索当前用户有权限的文档
只检索 2025 年之后的制度
只检索 product = "CRM" 的资料
```

检索逻辑可以理解为：

```text
先按元数据过滤候选集合
再在候选集合里做向量相似度搜索
```

没有 Metadata Filter 的企业知识库，容易出现越权回答。

## 9. 常见误区

### 误区一：向量数据库只保存向量

不对。

向量数据库通常还要保存 Chunk 原文和元数据，否则检索出来的向量无法用于回答。

---

### 误区二：有了向量数据库就不需要普通数据库

不对。

用户、权限、上传记录、任务状态、聊天历史，更适合普通关系型数据库。

---

### 误区三：只追求检索速度

RAG 的目标不是单纯快，而是：

- 找得准
- 来源清晰
- 权限正确
- 回答可信

速度只是其中一个指标。

## 11. 推荐实践

建议先用 pgvector 或一个简单本地向量库完成：

1. 创建文档表
2. 创建 Chunk 表
3. 保存 content、embedding、metadata
4. 根据问题向量检索 Top 5
5. 加上 knowledge_base_id 过滤条件

这个阶段不要急着做复杂 UI。先把“存进去、搜出来、带来源”跑通。
