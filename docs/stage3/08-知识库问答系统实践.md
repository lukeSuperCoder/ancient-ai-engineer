# 知识库问答系统实践

## 文档目标

在上一份实践（07）中，我们把《三国演义》入库到 PostgreSQL + pgvector。这份文档把知识库接入 Mini ChatGPT 应用，实现完整的 RAG 问答流程：

```text
用户提问
  -> 调用本地 qwen3-embedding:0.6b 获取查询向量
  -> pgvector 余弦相似度检索 Top-K 文本块
  -> 过滤低相似度结果（阈值可配置）
  -> 拼装 RAG Prompt（检索结果 + 用户问题）
  -> 调用 LLM 流式生成回答（标注出处、引用原文）
  -> SSE 流式返回浏览器
```

相关文件：

- `code/chatgpt/server/db.ts` — 数据库连接与向量检索
- `code/chatgpt/server/knowledge.ts` — RAG 问答流式逻辑
- `code/chatgpt/server/config.ts` — 环境变量配置
- `code/chatgpt/server/index.ts` — SSE 端点
- `code/chatgpt/src/composables/useChat.ts` — 前端知识库模式
- `code/chatgpt/src/components/ChatShell.vue` — KB 按钮
- `code/chatgpt/src/components/ChatInput.vue` — KB 模式提示

## 1. 整体架构

Mini ChatGPT 原有 Chat / Agent 两种模式。本次新增第三种模式 KB（Knowledge Base），三种模式共享同一套 SSE 流式通信架构，互不影响：

```text
┌─────────────────────────────────────────────┐
│              ChatShell.vue                   │
│  [ Chat ] [ Agent ] [ KB ]   ← 三模式切换    │
├──────────────────────┬──────────────────────┤
│                      │                      │
│   Chat → /api/chat/stream                  │
│   Agent → /api/agent/stream                │
│   KB → /api/kb/stream      ← 新增          │
│                      │                      │
├──────────────────────┴──────────────────────┤
│              server/index.ts                 │
│         writeSse() 统一 SSE 输出             │
└─────────────────────────────────────────────┘
```

KB 模式与 Chat 模式类似，只处理 `delta` 和 `error` 事件（不需要 Agent 模式的 thinking / step / structured 事件）。

## 3. 数据库连接与向量检索

`server/db.ts` 是新增的模块，职责单一：连接 PostgreSQL，提供向量检索能力。

### 3.1 连接池

使用 `pg` 库的 `Pool` 维持连接池，避免每次请求都新建连接：

```typescript
import { Pool } from "pg";
import { getModelConfig } from "./config";

const modelConfig = getModelConfig();

const pool = new Pool({
  host: modelConfig.dbHost,
  port: modelConfig.dbPort,
  user: modelConfig.dbUser,
  password: modelConfig.dbPassword,
  database: modelConfig.dbName,
});
```

`Pool` 内部管理多个 `Client`，请求结束后自动归还连接。

### 3.2 查询向量化

调用本地 Ollama 的 Embedding 接口，把用户问题转为 1024 维向量：

```typescript
export async function embedQuery(query: string): Promise<number[]> {
  const data = await fetchJson<EmbeddingResponse>(
    modelConfig.embeddingApiUrl,
    {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        model: modelConfig.embeddingModel,
        input: query,
      }),
    },
    "Embedding API 请求失败",
  );

  return data.data[0].embedding;
}
```

复用了项目已有的 `fetchJson<T>()` 工具函数（`server/tools/http.ts`），统一错误处理。

### 3.3 向量检索

SQL 查询与 Python 版本完全一致，使用 pgvector 的 `<=>` 余弦距离算子：

```sql
SELECT
  chunk_id       AS "chunkId",
  chapter_number AS "chapterNumber",
  chapter_title  AS "chapterTitle",
  chunk_index    AS "chunkIndex",
  content,
  1 - (embedding <=> $1::vector) AS similarity
FROM stage3_classic_chunks
WHERE work_id = 'classic_sanguo_yanyi'
ORDER BY embedding <=> $1::vector
LIMIT $2
```

检索后在代码层过滤低于阈值的结果：

```typescript
return rows.filter(
  (row: SearchResult) => row.similarity >= modelConfig.kbSimilarityThreshold,
);
```

阈值通过 `KB_SIMILARITY_THRESHOLD` 环境变量配置，默认 0.3。这个设计的原因：

- **在代码层而非 SQL 层过滤**：如果直接在 SQL 中加 `WHERE similarity >= 0.3`，当所有 Top-K 结果都低于阈值时，会返回空集；在代码层过滤可以同时保留"检索了但全被过滤掉"的语义，便于前端展示"没有找到"而非"查询出错"
- **阈值可调**：不同 embedding 模型的相似度分布不同，0.3 适合 qwen3-embedding:0.6b，换模型后可能需要调整

## 4. RAG 问答流式逻辑

`server/knowledge.ts` 是 KB 模式的核心，实现 `streamKnowledgeReply()` 异步生成器。

### 4.1 RAG Prompt 设计

这是整个知识库问答最关键的部分——如何让 LLM 忠实于检索结果、不胡编乱造：

```text
你是一个基于中国古典文学知识库的问答助手。请严格根据以下参考资料回答用户问题。

要求：
1. 如果参考资料中有相关内容，请基于资料回答，并在回答中标注出处（第X回 · 回目标题），引用原文关键片段。
2. 如果参考资料与问题无关或不足以回答问题，请直接说"知识库中没有找到相关内容，我无法回答这个问题。"
3. 不要编造任何不在参考资料中的信息。

参考资料：
【资料1】第1回 · 宴桃园豪杰三结义 斩黄巾英雄首立功
（原文内容...）

---

【资料2】第1回 · 宴桃园豪杰三结义 斩黄巾英雄首立功
（原文内容...）

用户问题：刘备关羽张飞在哪里结义？
```

三个核心约束：

| 约束 | 目的 |
|------|------|
| 必须基于参考资料回答 | 防止幻觉（hallucination） |
| 标注出处 + 引用原文 | 让答案可验证、可追溯 |
| 无关时说"不知道" | 避免牵强回答 |

### 4.2 无结果时的短路处理

如果检索结果为空（全部低于阈值），不调用 LLM，直接返回固定文案：

```typescript
if (results.length === 0) {
  yield "知识库中没有找到相关内容，我无法回答这个问题。";
  return;
}
```

这样做的意义：

- 节省一次 LLM 调用（省钱、省时间）
- 100% 避免 LLM 在无上下文时编造答案
- 给用户明确的信号：知识库确实没有相关内容

### 4.3 流式输出

有结果时，把检索到的文本块拼入 system prompt，只发送最后一条用户消息给 LLM，流式返回：

```typescript
const stream = client.messages.stream({
  model: modelConfig.model,
  max_tokens: modelConfig.maxTokens,
  system: ragPrompt,                    // RAG prompt 作为 system
  messages: [{ role: "user", content: query }],
});

for await (const event of stream) {
  if (event.type !== "content_block_delta" || event.delta.type !== "text_delta") {
    continue;
  }
  yield event.delta.text;
}
```

使用 Anthropic SDK 的 `stream()` 方法，只提取 `text_delta` 事件，与 Chat 模式的流式逻辑完全一致。

## 5. SSE 端点

在 `server/index.ts` 新增 `POST /api/kb/stream`：

```typescript
app.post("/api/kb/stream", async (request, response) => {
  response.setHeader("Content-Type", "text/event-stream; charset=utf-8");
  response.setHeader("Cache-Control", "no-cache, no-transform");
  response.setHeader("Connection", "keep-alive");
  response.flushHeaders();

  try {
    for await (const delta of streamKnowledgeReply(request.body)) {
      writeSse(response, "delta", { text: delta });
    }
    writeSse(response, "done", {});
  } catch (error) {
    const message = error instanceof Error ? error.message : "Unknown server error";
    writeSse(response, "error", { error: message });
  } finally {
    response.end();
  }
});
```

复用了已有的 `writeSse()` 函数，事件格式与 `/api/chat/stream` 完全一致（`delta` / `done` / `error`）。

## 9. 与 Agent 模式的区别

| 对比维度 | Agent 模式 | KB 模式 |
|---------|-----------|---------|
| 外部调用 | LLM 可能多次调用工具 | 先检索一次，再调用一次 LLM |
| 上下文来源 | 工具实时获取 | 预先入库的文本块 |
| 可靠性 | 工具可能失败/超时 | 数据固定，检索结果稳定 |
| 回答风格 | 自由发挥 | 严格基于检索结果 |
| 幻觉风险 | 有（模型可能编造工具结果以外的内容） | 低（prompt 约束 + 阈值过滤） |

KB 模式本质上是 RAG 的典型实现：检索与生成分离，用检索结果约束生成内容，是生产环境中最常用的知识问答架构。
