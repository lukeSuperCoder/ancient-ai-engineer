# 中国名著知识库入库实践

## 文档目标

这份实践文档把前面的 Embedding、Chunking、pgvector 串成一个真实案例：

```text
《三国演义》txt 文件
  -> 章节解析
  -> 文本切块
  -> 调用本地 qwen3-embedding:0.6b
  -> 写入 PostgreSQL + pgvector
  -> 向量检索
```

相关文件：

- `stage3/05_classic_literature_schema.sql`
- `stage3/05_ingest_three_kingdoms_embeddings.py`
- `stage3/05_search_classic_literature.py`

## 1. 表结构设计

本实践使用三张表：

```text
stage3_classic_works
  保存作品级信息，例如《三国演义》、作者、朝代、文本来源目录。

stage3_classic_chapters
  保存章节级信息，例如第几回、回目标题、原始文件路径、字符数。

stage3_classic_chunks
  保存 RAG 真正检索的文本块、原文偏移、metadata 和 embedding。
```

这样设计的原因：

- 作品表适合以后扩展到《西游记》《水浒传》《红楼梦》
- 章节表保留章回体小说的自然结构
- chunk 表面向向量检索，保存 embedding 和可追溯来源

## 2. 为什么使用 vector(1024)

本机接口：

```text
http://127.0.0.1:11434/v1/embeddings
model=qwen3-embedding:0.6b
```

实际返回 1024 维向量，所以 SQL 中使用：

```sql
embedding vector(1024)
```

pgvector 的维度必须和模型输出一致。

如果以后更换模型，需要同步修改：

- SQL 中的 `vector(1024)`
- 入库脚本中的 `EXPECTED_EMBEDDING_DIM`

## 3. 初始化数据库

先执行 schema：

```bash
psql -h localhost -p 5432 -U luke -d vector_db -f stage3/05_classic_literature_schema.sql
```

## 4. 试跑切块

建议先不请求模型、不写数据库，只看解析和切块结果：

```bash
python3 stage3/05_ingest_three_kingdoms_embeddings.py --limit-chapters 2 --dry-run
```

如果输出的章节数、chunk 数、预览内容正常，再进入真实入库。

## 5. 小批量入库

先入库前 2 回：

```bash
python3 stage3/05_ingest_three_kingdoms_embeddings.py --limit-chapters 2
```

确认没有维度错误、接口错误、数据库错误后，再执行全量：

```bash
python3 stage3/05_ingest_three_kingdoms_embeddings.py
```

## 6. 检索验证

入库后可以直接查询：

```bash
python3 stage3/05_search_classic_literature.py "刘备关羽张飞在哪里结义？"
```

返回结果会包含：

- `chunk_id`
- 第几回
- 回目标题
- 相似度
- metadata
- 原文片段

这就是后续 RAG 生成回答时需要拼进 Prompt 的上下文。
