# RAG Pipeline：检索增强生成

## 文档目标

这份文档讲清楚 RAG 的主流程：

> 用户提问后，系统如何检索资料、拼接 Prompt，并生成带依据的回答。

学完后，你应该能理解：

- Retrieval 是什么
- Prompt Assemble 是什么
- Generation 是什么
- RAG 的入库流程和问答流程
- 如何组织一个最小可用 RAG Pipeline

---

## 1. RAG Pipeline 分成两条链路

RAG 系统一般有两条核心链路。

### 1.1 文档入库链路

```text
上传文档
  ↓
解析文本
  ↓
清洗文本
  ↓
Chunk 切块
  ↓
生成 Embedding
  ↓
保存 Chunk、向量、元数据
```

这条链路解决：

> 如何把企业资料变成可检索的知识库。

### 1.2 用户问答链路

```text
用户提问
  ↓
问题生成 Embedding
  ↓
检索相关 Chunk
  ↓
组装 Prompt
  ↓
LLM 生成回答
  ↓
返回答案和来源引用
```

这条链路解决：

> 如何基于知识库回答用户问题。

---

## 2. Retrieval 是什么

Retrieval 就是检索。

它的任务是：

> 从知识库中找出和用户问题最相关的资料片段。

输入：

```text
用户问题：如何申请年假？
```

输出：

```text
Chunk 1：年假申请需要在 OA 系统提交。
Chunk 2：员工请假超过 3 天需要直属领导审批。
Chunk 3：年假余额可以在员工自助系统查询。
```

Retrieval 的质量直接影响最终回答。

如果检索结果错了，后面的模型再强也很难答对。

---

## 3. Prompt Assemble 是什么

Prompt Assemble 是把检索到的资料拼进模型输入。

一个典型 Prompt 可以这样组织：

```text
你是企业知识库助手。
请只根据下面的资料回答用户问题。
如果资料中没有答案，请说“知识库中没有找到相关依据”。
回答后列出引用来源。

【资料 1】
来源：员工手册.pdf，第 12 页
内容：年假申请需要在 OA 系统提交。

【资料 2】
来源：员工手册.pdf，第 13 页
内容：请假超过 3 天需要直属领导审批。

用户问题：
如何申请年假？
```

Prompt Assemble 的关键不是把资料都塞进去，而是：

- 只放相关资料
- 资料要有编号
- 来源要清楚
- 指令要约束模型不要乱答

---

## 4. Generation 是什么

Generation 是 LLM 基于 Prompt 生成最终回答。

例如：

```text
你可以在 OA 系统中提交年假申请。如果请假超过 3 天，需要直属领导审批。

引用来源：
- 员工手册.pdf，第 12 页
- 员工手册.pdf，第 13 页
```

这里要注意：

> LLM 的回答必须基于检索资料，而不是凭模型记忆。

如果资料里没有答案，应该让模型明确说明没有找到依据。

---

## 5. 最小 RAG 问答流程

可以用伪代码理解：

```ts
async function answerQuestion(question: string) {
  const queryEmbedding = await embed(question);

  const chunks = await vectorStore.search({
    embedding: queryEmbedding,
    topK: 5,
    filter: {
      knowledgeBaseId: "kb_hr"
    }
  });

  const prompt = buildRagPrompt({
    question,
    chunks
  });

  const answer = await llm.generate(prompt);

  return {
    answer,
    sources: chunks.map(chunk => chunk.source)
  };
}
```

这就是 RAG 的最小闭环。

## 6. Prompt 拼接原则

### 6.1 明确角色

```text
你是企业知识库助手。
```

### 6.2 限制回答范围

```text
只能根据给定资料回答。
资料中没有答案时，不要猜测。
```

### 6.3 保留来源编号

```text
【资料 1】
来源：xxx.pdf，第 3 页
内容：...
```

### 6.4 要求引用

```text
回答中需要标注引用来源。
```

## 7. 当检索不到资料时怎么办

不要强行让模型回答。

推荐返回：

```text
知识库中没有找到足够依据回答这个问题。
你可以尝试换一种问法，或上传相关资料后再提问。
```

工程上可以设置相似度阈值。

例如：

```text
如果 Top 1 相似度低于 0.35，则认为没有可靠资料。
```

具体阈值需要结合模型、数据和业务测试调整。

## 8. 多轮对话中的 RAG

多轮对话要注意一个问题：

> 用户当前问题可能依赖上一轮上下文。

例如：

```text
用户：年假怎么申请？
助手：...
用户：超过三天呢？
```

第二个问题“超过三天呢？”单独看不完整。

常见处理方式：

1. 把最近几轮对话一起交给模型改写成完整问题
2. 用改写后的问题做检索
3. 再基于检索资料生成回答

改写后可能是：

```text
员工申请年假超过三天时，需要什么审批流程？
```

## 9. 常见错误

### 错误一：把所有检索结果无脑塞进 Prompt

资料越多不一定越好。

不相关资料会干扰模型，让回答变得混乱。

---

### 错误二：没有告诉模型“资料不足就说不知道”

如果不加约束，模型可能根据常识补答案。

企业知识库里，这种补答案就是风险。

---

### 错误三：只返回答案，不返回来源

没有 Citation，用户无法判断答案是否可信。

企业场景里，来源引用是 RAG 的核心功能，不是装饰功能。

## 11. 推荐实践

建议实现一个最小 RAG Pipeline：

1. 手动准备 10 个 Chunk
2. 给每个 Chunk 生成 Embedding
3. 用户输入问题
4. 检索 Top 3 Chunk
5. 把 Chunk 拼进 Prompt
6. 让模型只根据 Chunk 回答
7. 返回答案和来源

先不要追求复杂架构。能把这条链路跑通，就已经掌握了 RAG 的核心。
