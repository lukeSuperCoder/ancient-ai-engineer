# 提示词工程六大策略详解

## 策略一：角色提示词 (Role Prompt)

### 核心思想
给模型一个明确的**身份**，让它从特定角度思考和回答。

### 模板
```
你是一个 [角色描述]，拥有 [能力描述]，
你的目标是 [目标描述]，
你的风格是 [风格描述]。
```

### 示例对比

#### ❌ 没有角色
```
用户：帮我写一个 Python 函数
```
→ 模型会给出泛泛的回答，风格不确定

#### ✅ 有角色
```python
system = """
你是一个拥有 10 年经验的 Python 高级开发工程师。
你遵循 PEP 8 编码规范，善于编写简洁、高效、可维护的代码。
你的回答风格：
- 先给出完整代码，再逐段解释
- 标注关键逻辑的注释
- 说明时间/空间复杂度
- 提供边界情况的处理
"""
```
→ 模型会给出专业、结构化的回答

### 角色设计要点
1. **具体优于笼统**："Python 数据工程专家" > "程序员"
2. **包含经验级别**："10 年经验" 影响回答深度
3. **定义风格**：让输出格式一致可预测
4. **设定目标**：让模型知道回答的最终目的

## 策略二：约束提示词 (Constraint Prompt)

### 核心思想
用明确的**规则和边界**限制模型的行为和输出格式。

### 模板
```
你必须遵守以下规则：
1. [规则1]
2. [规则2]
3. [规则3]

输出格式：
[格式模板]
```

### 示例

#### 格式约束
```python
system = """
你是一个代码审查助手。审查代码时必须遵循以下规则：

约束条件：
1. 每个问题必须标注严重等级：[严重] [警告] [建议]
2. 必须提供修复建议和修改后的代码
3. 不允许只说"代码没问题"，必须给出至少一个优化建议
4. 回复使用中文

输出格式：
## 审查结果
### [严重] 问题描述
- 位置：第 X 行
- 问题：...
- 修复：...
```python
修复后的代码
```
"""
```

#### 行为约束
```python
system = """
你是一个技术文档助手。严格约束：

绝对不允许：
- 编造不存在的 API 或函数
- 给出你不确定的答案（不确定时明确说明）
- 忽略用户的安全相关要求

必须做到：
- 所有代码示例必须可以直接运行
- 引用官方文档时标注来源 URL
- 每个回答结尾列出参考链接
"""
```

### 约束设计要点
1. **用否定句定义边界**："不允许做 X" 比 "应该避免做 X" 更有效
2. **用肯定句定义格式**：给出明确的输出模板
3. **约束要有验证方式**：让输出可以被程序检查
4. **优先级排序**：把最重要的约束放在最前面

## 策略三：Few-shot 示例 (Few-shot Prompting)

### 核心思想
通过给出**几个输入-输出示例**，让模型理解你期望的格式和行为。

### 模板
```
以下是一些示例：

输入：[示例1输入]
输出：[示例1输出]

输入：[示例2输入]
输出：[示例2输出]

现在请处理：
输入：[实际输入]
输出：
```

### 示例

#### 文本分类
```python
system = """
你是一个情感分析助手。根据用户文本判断情感倾向。

示例：
输入："这个产品太棒了，我非常喜欢！"
输出：{"情感": "积极", "置信度": 0.95, "关键词": ["棒", "喜欢"]}

输入："质量一般般，不值这个价格。"
输出：{"情感": "消极", "置信度": 0.80, "关键词": ["一般般", "不值"]}

输入："还可以吧，没什么特别的。"
输出：{"情感": "中性", "置信度": 0.70, "关键词": ["还可以", "没什么特别"]}
"""
```

### Few-shot 设计要点
1. **3-5 个示例最佳**：太少不够，太多浪费 Token
2. **示例要有代表性**：覆盖常见情况和边界情况
3. **示例要有多样性**：不要所有示例都太相似
4. **示例顺序有影响**：最近的示例权重更高（近因效应）
5. **示例格式一致**：让模型形成稳定的模式识别

## 策略四：思维链 (Chain of Thought, CoT)

### 核心思想
让模型**逐步推理**，而不是直接给出答案。这能显著提升复杂任务的准确率。

### 模板
```
请一步步思考：
1. 首先，[分析问题]
2. 然后，[制定计划]
3. 接着，[执行步骤]
4. 最后，[验证结果]
```

### 示例对比

#### ❌ 直接回答
```
用户：一个数组 [3,1,4,1,5,9,2,6] 的中位数是多少？
助手：中位数是 3.5
```
→ 可能算错，且无法验证

#### ✅ 思维链
```python
system = """
你是一个数学问题解决助手。解答时必须：
1. 先写出理解题目
2. 列出已知条件
3. 一步步推导
4. 验证结果
5. 给出最终答案

按照以下格式回答：
## 理解题目
[用自己的话描述问题]

## 解题步骤
步骤1：...
步骤2：...

## 验证
[检查答案是否合理]

## 答案
最终答案是：...
"""
```

### CoT 设计要点
1. **明确要求"逐步思考"**：这句话本身就是强大的 Prompt 技术
2. **提供推理框架**：给模型一个思考结构
3. **复杂任务必备**：数学、逻辑、代码调试等场景效果显著
4. **Zero-shot CoT**：仅仅加上"请一步步思考"就能提升效果
5. **可以和 Few-shot 结合**：示例中也展示推理过程

## 策略五：提示词模板 (Prompt Template)

### 核心思想
将提示词设计为**可参数化的模板**，通过变量注入实现复用和动态生成。

### 模板
```
你是一个 {role}，擅长 {skill}。

当前任务：{task}
上下文信息：{context}

输出要求：
{output_format}
```

### 代码实现
```python
class PromptTemplate:
    """提示词模板类 - 支持变量注入和模板复用"""
    
    def __init__(self, template: str, variables: dict = None):
        self.template = template
        self.variables = variables or {}
    
    def render(self, **kwargs) -> str:
        """渲染模板，将变量注入到模板中"""
        all_vars = {**self.variables, **kwargs}
        result = self.template
        for key, value in all_vars.items():
            result = result.replace(f"{{{key}}}", str(value))
        # 检查是否有未填充的变量
        import re
        unfilled = re.findall(r'\{(\w+)\}', result)
        if unfilled:
            raise ValueError(f"未填充的变量: {unfilled}")
        return result

# 使用示例
template = PromptTemplate(
    template="""
你是一个 {role}，拥有 {experience} 年经验。

当前任务：{task}
技术栈：{tech_stack}

请按照以下格式回答：
{output_format}
""",
    variables={
        "role": "Python 后端工程师",
        "experience": "8",
        "output_format": "## 方案\n[描述]\n\n## 代码\n```python\n[代码]\n```"
    }
)

# 渲染不同任务
result = template.render(
    task="设计一个用户认证系统",
    tech_stack="FastAPI + PostgreSQL + Redis"
)
```

### 模板设计要点
1. **变量用花括号标记**：`{variable_name}` 清晰直观
2. **区分固定部分和变化部分**：固定部分在初始化时设置
3. **支持默认值**：常用变量提供默认值，特殊场景覆盖
4. **验证完整性**：渲染后检查是否有未填充变量
5. **模板存储为文件**：`.md` 或 `.yaml` 文件，方便非技术人员编辑

## 策略六：动态上下文注入 (Dynamic Context)

### 核心思想
根据当前状态**动态生成**部分提示词内容，而非全部硬编码。

### 动态上下文的来源

```
时间信息：当前日期、星期、时区
环境信息：工作目录、操作系统、语言版本
用户状态：历史偏好、当前任务、权限级别
外部数据：数据库查询结果、API 响应、文件内容
```

### 示例
```python
import datetime
import os

def build_dynamic_context(user_preferences: dict = None) -> str:
    """
    构建动态上下文信息
    这部分内容每次调用都会重新生成
    """
    parts = []
    
    # 时间信息 - 让模型知道当前日期
    now = datetime.datetime.now()
    parts.append(f"当前时间：{now.strftime('%Y-%m-%d %H:%M:%S')}")
    parts.append(f"星期：{['一','二','三','四','五','六','日'][now.weekday()]}")
    
    # 环境信息
    parts.append(f"操作系统：{os.uname().sysname}")
    parts.append(f"工作目录：{os.getcwd()}")
    
    # 用户偏好（从记忆/配置加载）
    if user_preferences:
        prefs = []
        for k, v in user_preferences.items():
            prefs.append(f"- {k}: {v}")
        parts.append("用户偏好：\n" + "\n".join(prefs))
    
    return "\n".join(parts)
```

### 动态注入设计要点
1. **用分界线标记**：将动态部分与静态部分用明确的分界线分开
2. **动态部分不缓存**：每次请求重新生成
3. **信息要精简**：只注入必要的信息，避免 Token 浪费
4. **按需加载**：不同场景加载不同的上下文
5. **来源可追溯**：标注每条上下文的数据来源

## 六大策略的组合使用

在实际项目中，这些策略不是孤立使用的，而是**组合叠加**：

```
完整的 System Prompt = 角色提示词      (定义身份)
                     + 约束提示词      (定义规则)
                     + Few-shot 示例   (定义格式)
                     + 思维链框架      (定义推理方式)
                     + 提示词模板      (定义变量结构)
                     + 动态上下文      (注入运行时信息)
```

**关键原则**：每一层都让模型的行为更可预测、输出更可控。
