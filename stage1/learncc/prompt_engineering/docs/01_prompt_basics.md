# 提示词基础理论

## 1. 什么是 System Prompt？

System Prompt（系统提示词）是发送给大语言模型的**最高优先级指令**，它定义了模型在整个对话中的行为框架。

### 与其他提示词的区别

| 类型 | 位置 | 作用 | 优先级 |
|------|------|------|--------|
| System Prompt | 系统角色 | 定义身份、规则、约束 | 最高 |
| User Prompt | 用户角色 | 用户的实际请求 | 中 |
| Assistant Response | 助手角色 | 模型的回复 | 最低 |

### 为什么 System Prompt 如此重要？

```
没有 System Prompt 的模型 = "一个什么都能回答但什么都不精通的通才"

有了 System Prompt 的模型 = "一个有明确身份、规则和专业能力的高效助手"
```

System Prompt 的核心作用：
1. **定义身份（WHO）**：模型是谁？专家？助手？老师？
2. **定义能力（WHAT）**：模型能做什么？不能做什么？
3. **定义规则（HOW）**：模型应该遵循什么规则？
4. **定义格式（FORMAT）**：输出应该长什么样？

## 2. 提示词的组成结构

一个完整的系统提示词通常包含以下部分：

```
┌─────────────────────────────┐
│     Core Identity           │  ← 你是谁
├─────────────────────────────┤
│     Capabilities            │  ← 你能做什么
├─────────────────────────────┤
│     Constraints             │  ← 你不能做什么 / 必须怎么做
├─────────────────────────────┤
│     Available Tools         │  ← 你可以使用哪些工具
├─────────────────────────────┤
│     Memory / Context        │  ← 你需要记住什么
├─────────────────────────────┤
│     CLAUDE.md / Rules       │  ← 项目级 / 用户级规则
├─────────────────────────────┤
│ ═══ DYNAMIC BOUNDARY ═══    │  ← 静态/动态分界线
├─────────────────────────────┤
│     Dynamic Context         │  ← 每次变化的上下文
│     (日期、工作目录、模型等)   │
└─────────────────────────────┘
```

### 关键原则：稳定与动态分离

- **稳定部分**（缓存）：身份、规则、工具列表 → 跨会话不变
- **动态部分**（每轮重建）：日期、上下文、用户偏好 → 每次变化

这种分离的核心价值是**节省 Token / 降低成本 / 提升速度**。

## 3. 消息格式

大模型 API 使用消息列表来传递对话：

```python
messages = [
    # 系统提示词（最高优先级）
    {
        "role": "system",     # 或作为 API 的 system 参数
        "content": "你是一个专业的 Python 开发专家..."
    },
    # 用户消息
    {
        "role": "user",
        "content": "帮我写一个快速排序函数"
    },
    # 助手回复
    {
        "role": "assistant",
        "content": "好的，这是一个快速排序实现..."
    },
    # 后续对话...
    {
        "role": "user",
        "content": "能优化一下吗？"
    }
]
```

### 重要概念

1. **System 消息只能有一条**（通常在开头），定义全局行为
2. **User / Assistant 消息交替出现**，构成对话历史
3. **上下文窗口有限**（如 128K tokens），需要管理消息长度
4. **模型"记住"上下文的原因**：每次调用时，把完整的消息历史发给模型

## 4. Token 基础

### Token 是什么？

Token 是大模型处理文本的最小单位。大致规则：

```
1 个英文单词 ≈ 1-2 个 token
1 个中文字   ≈ 1-3 个 token
1 个代码符号 ≈ 1-2 个 token
```

### 为什么需要关心 Token？

```
Token 直接影响：
├── 成本（按 token 计费）
├── 速度（token 越多，响应越慢）
└── 上下文窗口（超过上限，模型会"遗忘"早期内容）
```

### System Prompt 的 Token 优化策略

1. **精简冗余**：去掉重复和废话
2. **稳定部分缓存**：不变的部分只算一次
3. **动态注入**：变化的上下文按需添加
4. **模板复用**：相同的指令不要写多遍

## 5. 关键参数

| 参数 | 作用 | 推荐值 |
|------|------|--------|
| `temperature` | 控制随机性。0=确定性，1=创造性 | 编程: 0, 写作: 0.7 |
| `top_p` | 核采样范围 | 0.9 |
| `max_tokens` | 最大输出长度 | 根据任务设定 |
| `stop_sequences` | 停止生成的标记 | 按需 |

**注意**：System Prompt 写得越好，对 `temperature` 的依赖就越低。好的提示词比调参更有效。
