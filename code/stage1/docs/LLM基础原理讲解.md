# LLM 基础原理讲解

## 文档目标

这份文档面向刚进入 AI 应用开发阶段的工程师，目标不是推公式，而是建立一套足够可靠的底层认知模型：

- LLM 到底如何“看懂”文本
- 为什么提示词、上下文、费用、响应速度都和 Token 有关
- 为什么 Embedding 能做语义检索
- 为什么 Transformer 能处理上下文
- 为什么不同采样参数会影响回答风格和稳定性

如果你已经会调用模型 API，但还不清楚模型内部大致如何工作，这篇文档就是补这块基础。

---

## 推荐学习顺序

理解 LLM，建议按下面这条链路学习：

1. Token 是什么
2. Embedding 是什么
3. Attention 是什么
4. Transformer 为什么能理解上下文
5. Context Window 是什么
6. 采样参数是什么：`temperature`、`top_p`、`max_tokens`

这条链路本质上对应了模型处理文本的流程：

`输入文本 -> 切分为 Token -> 转成向量 -> 建立上下文关系 -> 在上下文窗口内推理 -> 按采样策略输出`

## 一张图理解 LLM 的处理流程

```text
用户输入文本
   ↓
Tokenizer 分词
   ↓
Token ID 序列
   ↓
Embedding 向量化
   ↓
Transformer 计算上下文关系
   ↓
预测下一个 Token 的概率分布
   ↓
采样参数决定最终输出
   ↓
生成文本
```

## 1. Token 是什么

### 1.1 定义

Token 不是“字”，也不一定是“单词”，它是模型处理文本时使用的最小离散单位。

例如，文本：

```text
Hello world
```

可能会被切成：

```text
["Hello", " world"]
```

中文也不一定按单字切分，例如：

```text
你好世界
```

可能被切成：

```text
["你", "好", "世界"]
```

也可能根据 tokenizer 规则被切得更细或更粗。

### 1.2 为什么 Token 很重要

因为模型并不直接理解“文字”，模型真正处理的是数字编号，也就是 Token ID。

例如：

```text
"你好" -> [2312, 8871]
```

模型看到的不是“你好”这两个字，而是一串数字序列。后续的所有计算，都是围绕这些编号展开。

### 1.3 工程上你必须理解的几点

- Prompt 长度按 Token 计，不按字符数计
- 费用通常按输入 Token 和输出 Token 计费
- 上下文窗口限制的本质是 Token 总数限制
- 中文、代码、JSON 往往比自然英文更容易消耗 Token

### 1.4 常见误区

- 误区一：一个汉字一定等于一个 Token
- 误区二：一个英文单词一定等于一个 Token
- 误区三：字符数差不多，Token 数也差不多

这些都不成立。不同语言、空格、标点、缩写、代码格式，都会影响切分结果。

### 1.5 学完后的检查标准

学完 Token 之后，你至少应该能回答：

- 为什么模型厂商的价格表按 Token 计费
- 为什么一段很长的 JSON 往往比同长度自然语言更贵
- 为什么聊天历史越长，调用成本越高

### 1.6 推荐实践

用 tokenizer 工具亲自测试以下内容的 Token 差异：

- 一句中文
- 一句英文
- 一段 JSON
- 一段代码

推荐工具：

- OpenAI Tokenizer: https://platform.openai.com/tokenizer

## 2. Embedding 是什么

### 2.1 定义

Embedding 的本质，是把 Token、词、句子或整段文本表示成一组高维向量。

例如：

```text
"猫" -> [0.12, -0.44, 0.91, ...]
"狗" -> [0.13, -0.40, 0.88, ...]
```

神经网络不能直接处理文字，只能处理数字，因此文本必须先被转换成可计算的向量表示。

### 2.2 为什么 Embedding 有用

Embedding 不只是“数字化”，更关键的是它会尽量保留语义关系。

理想情况下：

- 语义相近的词或句子，在向量空间里更接近
- 语义差异大的词或句子，在向量空间里更远

例如：

- “猫”和“狗”通常比“猫”和“飞机”更接近
- “退款流程”与“售后政策”通常比“退款流程”与“天气预报”更接近

### 2.3 Embedding 和 LLM 的关系

需要区分两种场景：

- 在生成模型内部，Token 会先被映射为向量，作为模型输入
- 在应用工程里，我们也常单独调用 embedding 模型，把文档或问题转成向量做检索

前者服务于“生成”，后者服务于“语义搜索、召回、匹配”。

### 2.4 工程价值

Embedding 是下面这些能力的基础：

- RAG 中的向量检索
- 文档相似度搜索
- 语义去重
- 问答匹配
- 推荐系统中的语义召回

### 2.5 常见误区

- 误区一：Embedding 就是关键词匹配
- 误区二：两个文本字面不一样，就一定匹配不到
- 误区三：Embedding 可以直接替代生成模型

Embedding 更适合“找相关内容”，不适合直接完成复杂推理或生成任务。

### 2.6 学完后的检查标准

你应该能说清楚：

- 为什么 RAG 首先要做向量化
- 为什么 Embedding 能实现“语义相似”，而不只是字符串相同
- 为什么向量数据库适合存储文档片段表示

### 2.7 推荐实践

可以做一个小实验：

1. 准备几条意思相近和意思不相近的句子
2. 把它们转成 Embedding
3. 用余弦相似度比较距离
4. 观察“字面不同但语义接近”的句子是否会靠得更近

## 3. Attention 是什么

### 3.1 一句话定义

Attention 的作用是：让模型在处理当前 Token 时，动态决定应该重点关注输入序列中的哪些部分。

### 3.2 直觉理解

看这个句子：

```text
The animal didn't cross the street because it was too tired.
```

当模型处理 `it` 时，关键问题是：`it` 指的是谁？

更可能是：

- `animal`

而不是：

- `street`

Attention 机制做的事情，就是为“当前词”和“前面所有词”计算相关性权重，再用这个权重聚合信息。

### 3.3 本质上在做什么

可以粗略理解为三步：

1. 当前 Token 去看序列中其他 Token
2. 计算“我和谁更相关”
3. 按相关程度加权汇总信息

所以 Attention 不是简单记顺序，而是在做“关系建模”。

### 3.4 为什么它比传统 RNN 更强

传统 RNN 按顺序一个一个读：

```text
前一个状态 -> 当前状态 -> 下一个状态
```

问题在于：

- 信息传递链条长
- 长距离依赖容易丢失
- 并行计算能力弱

Attention 则允许当前位置直接关注远处信息，不必把所有信息都压缩进一个短状态里。

### 3.5 工程上你要理解什么

- Attention 让模型更擅长处理长距离依赖
- 它是 Transformer 能理解上下文的核心机制
- 生成每个 Token 时，模型并不是“拍脑袋”，而是在已有上下文中做条件概率预测

### 3.6 学完后的检查标准

你应该能解释：

- 为什么 Attention 能解决部分长程依赖问题
- 为什么模型在生成某个词时需要“回看”前文
- 为什么现代 LLM 的核心架构建立在 Attention 上

## 4. Transformer 为什么能理解上下文

### 4.1 核心结论

Transformer 的上下文理解能力，主要来自两部分：

- Self-Attention
- Positional Encoding 或位置相关机制

### 4.2 Self-Attention 的作用

Self-Attention 表示：序列中的每个位置都可以与同一序列中的其他位置建立关系。

例如句子：

```text
我 爱 北京 天安门
```

当模型处理“天安门”时，它不只是看前一个词，而是可以直接关联：

- 我
- 爱
- 北京

也就是说，它不是逐字机械推进，而是在整段上下文中建立联系。

### 4.3 为什么还需要位置信息

如果只有 Attention，没有位置信息，那么：

```text
我打你
你打我
```

在词集合相同的情况下，模型很难区分顺序差异。

所以 Transformer 需要额外引入“位置”概念，让模型知道：

- 哪个词在前
- 哪个词在后
- 谁和谁的相对距离更近

### 4.4 为什么长文本更贵

Attention 的一个重要代价是计算复杂度通常接近 `O(n^2)`。

这里的 `n` 是序列长度。当上下文变长时，任意两个位置之间都可能要计算关系，因此计算量和显存占用都会明显上升。

这就是为什么：

- 长上下文模型更贵
- 长文本推理更慢
- 聊天历史越多，延迟越高

### 4.5 学完后的检查标准

你应该能说清楚：

- Transformer 不是靠“记忆全文”，而是靠注意力机制动态建模关系
- 为什么位置编码是必要的
- 为什么上下文窗口越大，成本和延迟通常越高

## 5. Context Window 是什么

### 5.1 定义

Context Window 指的是模型单次调用时最多能够接收和利用的 Token 总量。

例如：

```text
128k context window
```

表示模型一次最多处理大约 12.8 万个 Token 的上下文。

### 5.2 上下文窗口包含什么

它不只是用户这一次输入的内容，通常还包括：

- system prompt
- 历史对话
- 检索返回的文档片段
- 工具调用结果
- 当前用户消息
- 模型正在生成的输出空间预留

所以“我只发了一小段文字，为什么还是超长”这种情况很常见，因为真正被送入模型的是全部拼接后的上下文。

### 5.3 超出窗口会怎样

超出后通常会发生以下情况之一：

- 较早内容被截断
- 中间部分被压缩或摘要
- 请求直接报错

这取决于具体平台和应用设计。

### 5.4 工程意义

Context Window 直接影响：

- 聊天系统的记忆长度
- RAG 中 chunk 的切分策略
- Agent 的历史记录保留方式
- Prompt 设计是否需要压缩
- 每轮调用的延迟和成本

### 5.5 为什么聊天越聊越慢

因为多数聊天应用并不是只发送“最后一句”，而是会把一部分历史消息连同当前问题一起重新提交给模型。

也就是说，模型每次回答时都会重新阅读当前保留的上下文。

### 5.6 学完后的检查标准

你应该能回答：

- 为什么一个会话越长越贵
- 为什么 RAG 文档切块不能无限大
- 为什么 Agent 系统必须做上下文管理

## 6. 采样参数：Temperature / Top_p / Max Tokens

模型在每一步生成下一个 Token 时，本质上会得到一个概率分布。采样参数决定了模型如何从这个分布中选出最终结果。

### 6.1 Temperature

`temperature` 用来控制输出的随机性。

可以粗略理解为：

- 低温：更保守，更稳定，更倾向高概率答案
- 高温：更发散，更多样，更容易出现创造性表达，也更容易跑偏

常见经验：

- `0 ~ 0.3`：适合代码生成、结构化抽取、严谨问答
- `0.5 ~ 0.8`：适合常规对话
- `0.8+`：适合创意写作、头脑风暴

### 6.2 Top_p

`top_p` 表示只从累计概率达到某个阈值的候选集合中采样。

例如：

- `top_p = 0.9`

可以理解为：只在“累计概率前 90%”的候选 Token 范围内随机选择。

它的作用也是控制发散程度，但方式不同于 temperature。

### 6.3 Temperature 和 Top_p 的关系

两者都影响采样随机性，因此工程实践里通常：

- 先固定一个
- 再调另一个

不要一开始同时大幅改两个参数，否则很难判断到底是哪一个导致输出变化。

### 6.4 Max Tokens

`max_tokens` 或同类参数用于限制模型最多生成多少 Token。

它的作用包括：

- 控制响应长度
- 限制费用
- 防止无意义长输出
- 为上下文窗口留出空间

### 6.5 工程建议

- 做抽取、分类、代码生成时，优先使用较低 temperature
- 做创意生成时，再适当提高 temperature 或调整 top_p
- 生产系统里必须给输出长度加上限

### 6.6 学完后的检查标准

你应该能解释：

- 为什么同一个 Prompt 每次输出可能不同
- 为什么低 temperature 更适合稳定任务
- 为什么必须限制生成长度

## 7. 把这些概念串起来理解

现在把前面的概念合在一起，形成完整流程：

### 7.1 输入阶段

用户输入一段文本，首先被 tokenizer 切分成 Token。

### 7.2 表示阶段

每个 Token 被映射为向量表示，也就是 embedding。

### 7.3 建模阶段

Transformer 使用 attention 机制分析这些 Token 之间的关系，并结合位置信息理解上下文。

### 7.4 约束阶段

模型只能在 context window 允许的范围内使用上下文。

### 7.5 输出阶段

模型在每一步都预测“下一个 Token 的概率分布”，再根据 `temperature`、`top_p`、`max_tokens` 等参数进行采样，逐步生成最终文本。

## 8. 工程师视角下最重要的几个结论

如果你是应用开发者，不需要一开始就深钻公式，但下面这些结论必须牢固：

### 8.1 LLM 本质上是下一个 Token 预测器

它不是先“想明白整段答案”再一次性输出，而是逐步预测下一个 Token。

### 8.2 模型处理的是 Token，不是原始文本

所以成本、长度限制、速度和 Token 强相关。

### 8.3 Embedding 适合“找内容”，生成模型适合“用内容”

RAG 的典型流程就是：

`先用 embedding 找相关文档 -> 再把文档交给生成模型回答`

### 8.4 上下文不是无限记忆

模型没有你想象中的长期记忆。它主要依赖你当前这轮传入的上下文。

### 8.5 采样参数会直接影响系统稳定性

很多“模型怎么今天不稳定”的问题，本质不是模型坏了，而是采样策略不适合当前任务。

## 9. 常见误区汇总

### 误区一：模型理解的是自然语言本身

更准确地说，模型处理的是 Token 和向量表示。

### 误区二：上下文窗口越大，模型就一定越聪明

窗口更大只表示“能看更多内容”，不等于推理能力必然更强。

### 误区三：Embedding 就是记住关键词

Embedding 做的是语义空间表示，不只是字面匹配。

### 误区四：Temperature 越低越好

低 temperature 适合稳定任务，但创意任务可能需要更高的多样性。

### 误区五：聊天记录天然会被模型永久记住

多数系统里，所谓“记忆”只是应用层把历史对话再次塞回上下文。

## 11. 最适合工程师的学习方法

不要只看理论，建议采用“每学一个概念，立刻做一个实验”的方式。

### 建议配套实验

- Token：比较中文、英文、JSON、代码的 Token 数

- Attention：看 attention 可视化示例
- 采样参数：固定 Prompt，分别修改 `temperature` 和 `top_p`
- Context Window：在长对话里观察延迟、费用和回答质量变化

### 为什么这种方式有效

因为 LLM 相关概念都很容易“听懂但没真正理解”。只有自己做过实验，才会把抽象概念转成工程直觉。

## 14. 总结

LLM 的核心并不神秘，可以先抓住下面这条主线：

- 文本先被切成 Token
- Token 被转成向量
- Transformer 用 Attention 理解上下文关系
- 模型在上下文窗口内逐步预测下一个 Token
- 采样参数决定输出是更稳定还是更发散

当你把这条链路真正理解清楚，再去学 Prompt、RAG、Agent，会轻松很多，因为你已经知道这些上层技术到底是在和模型的哪一层能力打交道。
